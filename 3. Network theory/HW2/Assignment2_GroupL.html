Analysis
Our main objective is to identify who the leaders and opinion-makers are in the cohort.

Step 1: Data pre-processing & graph created
First, we downloaded the sent and received tables and replaced all missing values with zero (to signify no messages received or sent). We expected the transpose of the received text matrix to equal the sent matrix, given the assumption that every message recorded as sent was recorded as received by the relevant student. These matrices however did not match up simply because of inaccurate/imprecise recordings. We chose to retain information from both matrices in cohort_mat by averaging the values of the T(received) and sent. Using cohort_mat we created a directed graph with 81 nodes and 2582 edges.

Step 2: Centrality measures calculated
We calculated and recorded the following centrality measures for each node: degree, closeness, pagerank and betweeness. We also calculated the local clustering coefficients for the underlying undirected graph. Below is a snapshot of each nodes centrality measures.

This table reveals many interesting insights. First of all, none of the nodes with the greatest ten clustering coefficients are featured in the other top ten centrality columns (i.e. nodes with high clustering coefficients tend not to have other high centrality measures). This reaffirms the idea that hubs often have surprisingly low clustering coefficients as their edges reach out to multiple distinct communities. Nodes with high clustering coefficients tend to be embedded in dense subgraphs meaning that their overall influence or importance on the network is less significant. For example, node 16 has the highest clustering coefficient (1.0) which could be misinterpreted as representing its high importance in the network. We know from the plot above that node 16 is incredibly isolated and belongs to a small clique of people.
If we now focus on the other centrality measures we see that node 32 has the highest degree, closeness and betweeness centralities; however, it does not feature in the top ten pageranked nodes. This suggests that node 32 is highly connected and central in the network topology however it is likely connected to less important people (people who are less connected) due to its low pagerank. In the real world, this could be someone who can reach the outskirts of the network i.e. communicate to people who typically communicate with a small sub network. Node 22 on the other hand has the highest page rank and the second highest degree, closeness and betweeness. This suggests it is central, well connected and connected to relatively important people (pagerank). On the back of this, node 22 looks to be the most important in the network.
Another node of interest is 73. It has a very high page rank and betweeness measure but a relatively lower degree and closeness centrality. This means that it has fewer number of connections (degree) and is less central (closeness); but it is an important bridge between many pairs of nodes (betweeness) and the connections it does have are with well- connected nodes (page rank).

#distance based on Kilometer
def distance_on_unit_sphere(lat1, long1, lat2, long2):

    degrees_to_radians = math.pi/180.0
 
    phi1 = (90.0 - lat1)*degrees_to_radians
    phi2 = (90.0 - lat2)*degrees_to_radians
 
    theta1 = long1*degrees_to_radians
    theta2 = long2*degrees_to_radians

    cos = (math.sin(phi1)*math.sin(phi2)*math.cos(theta1 - theta2) +
    math.cos(phi1)*math.cos(phi2))
    arc = math.acos(cos)
 
    return arc*6373 #km

def subtour_eliminate(model, where):
    if where == GRB.callback.MIPSOL:
        selected = []
    # make a list of edges selected in the solution
        for start in range(n):
            sol = model.cbGetSolution([model._vars[start,end] for end in range(n)])
            selected += [(start,end) for end in range(n) if sol[end] > 0.5]
    # find the shortest cycle in the selected edge list
        tour = subtour(selected)
        if len(tour) < n:
    # add a subtour elimination constraint
            expr = 0
            for start in range(len(tour)):
                for end in range(start+1, len(tour)):
                    expr += model._vars[tour[start], tour[end]]
            model.cbLazy(expr <= len(tour)-1)

def subtour(edges):
    visited = [False]*n
    cycles = []
    lengths = []
    selected = [[] for i in range(n)]
    for x,y in edges:
        selected[x].append(y)
    while True:
        current = visited.index(False)
        thiscycle = [current]
        while True:
            visited[current] = True
            neighbors = [x for x in selected[current] if not visited[x]]
            if len(neighbors) == 0:
                break
            current = neighbors[0]
            thiscycle.append(current)
        cycles.append(thiscycle)
        lengths.append(len(thiscycle))
        if sum(lengths) == n:
            break
    return (cycles[lengths.index(min(lengths))])

class Graph(object):
    def __init__(self):
        self.edges = dict() # empty dict of edge weights
        self.vertices = set() # emtpy set of vertices
        
    def add_edge(self, vertex1, vertex2, weight):
        self.edges[(vertex1, vertex2)] = weight 
        self.vertices.add(vertex1)
        self.vertices.add(vertex2)
        
    def add_vertex(self, vertex):
        self.vertices.add(vertex)
    
    def print_edges(self):
        for k,v in self.edges.items():
            print(k, v)
            
    def print_vertices(self):
        for v in self.vertices:
            print(v)
            
    def get_edges(self):
        return self.edges
    
    def get_vertices(self):
        return self.vertices
    
    def sort_by_weight(self):
        sorted_edges = []
        for all_edges in sorted(self.edges, key = self.edges.get):
            sorted_edges.append([all_edges, self.edges[all_edges]])
        return sorted_edges

parent = dict()
rank = dict()

def print_parent():
    for k,v in parent.items():
        print(k, v)
    
def print_rank():
    for k, v in rank.items():
        print(k, v)
        
def makeset(vertice): # constant time
    parent[vertice] = vertice
    rank[vertice] = 0

# Returns parent of the vertice to be found
def find(vertice):
    if parent[vertice] != vertice:
        parent[vertice] = find(parent[vertice])
    return parent[vertice]
    
def union(x, y):
    ''' make root of shorter tree point to root of taller tree
    use rank of vertices to determine height of tree'''
    root_x = find(x)
    root_y = find(y)
    if root_x == root_y:
        return
    if root_x != root_y:
        if rank[root_x] > rank[root_y]:
            parent[root_y] = root_x
        else:
            parent[root_x] = root_y
        if rank[root_x] == rank[root_y]:
            rank[root_y] += 1
    
    
def kruskal(graph):
    u = graph.get_vertices()
    for vertex in u:
        makeset(vertex)
    
    # Minimum Spanning Tree
    X_edges = set()
    sorted_weights = graph.sort_by_weight() # O(log m) 
    for edge_weight in sorted_weights: # O(m)
        node1, node2 = edge_weight[0]
        weight = edge_weight[1]
        if find(node1) != find(node2):
            X_edges.add(((node1, node2), weight))
            union(node1, node2) # O()
    
    return X_edges

The greedy approach to finding a minimum or maximum spanning tree in a graph with n nodes and m edges follows these steps:

  1. Order all edges by weight (increasing weight for minimum spanning tree, decreasing weight for maximum spanning tree).
  2. For each ordered edge, add it to the tree as long as it does not create a cycle.
  3. Continue until all edges are processed.

Sorting the edges in order of weight takes $O(m log m)$ time. Next, the main loop of the algorithm iterates until all the edges have been processed, which takes $O (m)$ time. Since all other operations run inside this loop, the algorithm runs in finite time, and ends onces all the edges have been processed.

Inside this main loop, there are two operations, $find$ and $union$, to check whether an edge will create a cycle if it is added to the spanning tree.

To determine if adding the next edge will create a cycle, the algorithm checks whether the edge links two parts of a cut. By definition, a cut is any partition between two sets of vertices, and adding an edge (lightest or heaviest, depending on whether the algorithm is finding a maximum or minimum spanning tree) from the list of ordered edges between any cut is guaranteed not to create a cycle if there are no edges already existing between the two partitions of the cut. Initially, each vertex of the graph is assigned to a set, which takes $O(1)$ time, and each time a new edge is processed, its endpoints are checked to see whether they belong to the same set or different sets. This operation, called $find$, takes $O (n)$ time. The new edge can be added to the spanning tree if each of its nodes are part of different partitions of the cut. If they are part of the same partition, they are already part of a tree and their addition would create a cycle.

If both endpoints of the new edge are in different partitions, the edge links two parts of a cut, and the new edge is added to the spanning tree's edges. Furthermore, the edge's endpoints are linked through a $union$ operation. A union stores each set of nodes as a tree structure, with each additional node being assigned as the child of an existing node in that set. When two sets of nodes are merged by a union, the smallest tree's root node points to the root of the largest tree. This operation takes $O (n)$ time.

Overall, the running time of the algorithm is $O m(log m)$ for sorting, $O (m)$ to loop through and process each edge, and within that loop, $O (n)$ to find whether two nodes belong to the same set, and $O (n)$ to merge two set of nodes. Overall, the upper bound of the algorithm's running time is therefore $O n^2$, which is polynomial running time.
